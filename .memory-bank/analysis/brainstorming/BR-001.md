# Brainstorming Report - BR-001

## Status

- Date: 2026-06-18
- Facilitator: AI
- Source of product intent: user
- Recommended next step: /brief

## Problem Space

MVP интернет-магазина должен провести покупателя от просмотра товара до оплаченного заказа без лишней сложности. Основной продуктовый контур: каталог, корзина без регистрации, OAuth-вход, checkout с обязательным телефоном, оплата через ЮKassa и создание заказа в Medusa Admin.

## Goals

- Быстро собрать рабочий MVP интернет-магазина на Medusa v2 и Next.js.
- Не модифицировать Medusa Core.
- Сохранить простую архитектуру: API -> Workflows -> Modules.
- Оформлять внешние интеграции отдельными модулями.
- Поддержать гостевую корзину с последующей привязкой к пользователю.
- Довести заказ до статуса, видимого в Medusa Admin, только после успешной оплаты.

## Constraints

- Backend: Medusa v2, TypeScript, PostgreSQL.
- Frontend: Next.js, TypeScript.
- Auth: Google OAuth, VK ID.
- Payments: ЮKassa, карты, СБП, SberPay.
- Local development: Docker Compose.
- KISS: без микросервисов, enterprise-функций и расширений вне MVP.
- Не входят в MVP: 1С, СДЭК/Boxberry, бонусы, маркетплейс, B2B, мобильное приложение, SMS-подтверждение.

## Techniques Used

- Constraints-first framing: сначала зафиксированы стек, архитектурные правила и исключения.
- MVP slicing: выделен минимальный путь от выбора товара до оплаченного заказа.
- Risk-first review: отмечены места, где интеграции и состояния могут нарушить простой поток.
- Journey clustering: идеи сгруппированы вокруг каталога, корзины, checkout, оплаты и операционного отображения заказа.

## Raw Ideas

- Сделать основной сценарий "гость выбирает товар -> кладет в корзину -> оплачивает -> получает заказ".
- Отложить личный кабинет до момента, когда он нужен для заказа или истории заказов.
- Хранить гостевую корзину так, чтобы она переживала перезагрузку страницы и новую сессию.
- При OAuth-входе явно объединять или привязывать гостевую корзину к customer profile.
- Считать телефон обязательным checkout-полем независимо от OAuth-провайдера.
- Начать с базовых способов доставки без интеграций с внешними службами.
- Не смешивать оплату и создание заказа, пока webhook не подтвердил успешный платеж.
- Сделать платежный модуль ЮKassa отдельной границей интеграции.
- Сохранять payment status так, чтобы повторный webhook не создавал дубли заказов.
- Сделать страницу успешной оплаты устойчивой к задержке webhook.
- Считать Medusa Admin операционным интерфейсом для MVP, не строить отдельную админку.
- Начать поиск и фильтрацию с простого server/API-backed варианта без отдельного search engine.
- Зафиксировать, какие product/category fields нужны для MVP-витрины.
- Свести frontend к storefront flow, не создавать маркетинговую landing page как основной экран.
- Добавить локальный Docker Compose как обязательный developer path.
- Оставить 1С, доставочные интеграции, бонусы, B2B и marketplace как явные future directions, не как скрытые hooks.
- Оптимизировать каталог под товары для дома, включая разные виды карнизов.
- Включить категории и фильтры, но ограничить их умеренным набором для MVP.
- Рассмотреть email-уведомления о заказе как MVP-добавление с высокой ценностью.
- Рассмотреть wishlist/избранное как новую идею, если она не раздует первый релиз.
- Включить MVP-фильтры: категория, цена, цвет, материал, размер/длина, тип товара, способ крепления.
- При логине объединять гостевую и пользовательскую корзины, одинаковые позиции суммировать.
- Делать wishlist/избранное только для авторизованных пользователей.
- Отправлять email для pending-заказа, успешной оплаты, ошибки оплаты и изменения статуса.
- Для pending-заказов дать retry оплаты и автоматическое истечение/отмену после таймаута.
- Моделировать товар как одну карточку с вариантами/SKU по цвету, длине, размеру и материалу.
- Устанавливать 72-часовой таймаут для pending-заказа без оплаты.
- Резервировать товарные остатки на время pending-заказа.
- Показывать способы доставки: самовывоз, курьер по городу, доставка транспортной компанией без интеграции.
- Использовать фиксированные тарифы доставки по выбранному способу.
- Собирать в checkout город, адрес, комментарий и выбранный способ доставки.
- Считать webhook ЮKassa главным источником статуса оплаты; return page показывает ожидание или результат.
- Обрабатывать повторные webhook-события идемпотентно.
- Не реализовывать фискализацию/чеки в MVP, но записать это как legal/payment risk перед запуском.

## Clusters / Themes

- Purchase journey: каталог, карточка товара, корзина, checkout, платеж, заказ.
- Identity and cart continuity: guest cart, OAuth login, cart ownership transfer.
- Payment correctness: ЮKassa redirect/payment methods, webhook, idempotency, order creation after success.
- Medusa extension boundaries: workflows/modules instead of core changes.
- MVP discipline: no delivery-provider integrations, no loyalty, no B2B, no mobile app.
- Local operability: Docker Compose with PostgreSQL and clear startup flow.

## Selected Directions

- Use `/brief` next to convert the user idea and this brainstorming report into a concise product brief.
- Keep MVP centered on one complete purchase path, not on account/admin expansion.
- Treat ЮKassa payment status handling and idempotent order creation as a core risk area for later PRD/spec work.
- Treat guest cart persistence and post-login attachment as a core product behavior for later PRD/spec work.
- Keep delivery as selectable MVP options without external carrier integration.
- Target catalog domain: товары для дома, включая разные виды карнизов.
- Catalog scope: categories and filters are included, but kept moderate rather than broad enterprise catalog search.
- Auth scope: Google OAuth and VK ID are both included in MVP.
- Checkout identity: guest cart is allowed, but login is required before payment.
- Delivery scope: simple manual delivery options are included; no external delivery-provider integration in MVP.
- Order lifecycle direction: create order before payment with a pending status, then update after ЮKassa payment status changes.
- New idea selected for MVP consideration: email order notification.
- New idea selected for further consideration: wishlist/favorites.
- Catalog filters direction: category, price, color, material, size/length, product type, mounting method.
- Product variants/SKU direction: one product card with selectable variants/SKU by color, size, length, and material.
- Cart merge direction: merge guest and existing user carts on login; sum identical items.
- Wishlist direction: authenticated users only.
- Email notification direction: pending order, payment success, payment failure, and status change.
- Order status direction: `pending_payment -> paid -> processing -> completed/canceled/refunded`.
- Pending order direction: allow payment retry and expire/cancel stale pending orders after a timeout.
- Admin visibility direction: contact data, products, delivery, payment status, order status, total, and payment method.
- Pending payment timeout direction: 72 hours.
- Inventory direction: reserve stock while an order is in pending payment.
- Delivery options direction: pickup, city courier, and transport-company delivery without external integration.
- Delivery pricing direction: fixed tariffs per delivery method.
- Delivery data direction: city, address, comment, and selected delivery method.
- Payment status direction: ЮKassa webhook is the source of truth; return page only reflects waiting/result state.
- Webhook robustness direction: repeated webhook events must be idempotent.
- Fiscalization direction: do not implement receipts/fiscalization in MVP, but keep it as a legal/payment launch risk.

## Rejected Directions

- Do not start task decomposition from the brainstorm output.
- Do not create Product Brief or PRD inside `/brainstorm`.
- Do not add 1С, СДЭК/Boxberry, bonuses, marketplace, B2B, mobile app, or SMS confirmation to MVP.
- Do not introduce microservices or enterprise architecture patterns.
- Do not modify Medusa Core.

## Assumptions

- `idea.md` is the current source of user intent.
- The MVP is for a Russian-market checkout context because ЮKassa, СБП, SberPay and VK ID are included.
- A simple delivery method selector is acceptable for MVP until a later delivery integration is explicitly requested.
- Medusa Admin is sufficient for order operations in the first MVP.
- No external research was performed during brainstorming.
- Fiscalization/receipts are not implemented in MVP and remain a legal/payment launch risk before production.
